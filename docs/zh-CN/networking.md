# 网络与代理说明

当前版本专门处理了一类常见问题：

- 不开代理时，搜索页访问不稳定，表现为“找不到资源”
- 开代理后，搜索能成功，但播放器拉取播放流失败

这往往不是代码逻辑错误，而是“搜索站点”和“视频 CDN”对网络出口的要求不同。

## 常见现象

以中文站点为例，通常会涉及两类域名：

- 搜索页、详情页域名
- 实际播放流或分片的 CDN 域名

可能出现的情况是：

- 搜索页更适合走代理
- 视频流更适合直连

于是就会出现：

- 不开代理时搜索失败
- 开代理后播放器无法播放分片或 `m3u8`

## 当前处理方式

当前版本默认引入了“播放器尽量直连”的策略。

也就是说：

- `ani-cli` 自身的搜索与页面解析仍然继承你当前终端环境
- 真正启动播放器时，默认会清掉常见代理环境变量

涉及的环境变量包括：

- `http_proxy`
- `https_proxy`
- `HTTP_PROXY`
- `HTTPS_PROXY`
- `all_proxy`
- `ALL_PROXY`
- `no_proxy`
- `NO_PROXY`

## 默认行为

默认情况下：

```bash
ANI_CLI_PLAYER_NO_PROXY=1
```

这表示播放器启动时尽量走直连。

如果你想保留播放器继续走代理，可以显式关闭这个行为：

```bash
ANI_CLI_PLAYER_NO_PROXY=0 ani-cli --source omofun111 "进击的巨人"
```

## 推荐排查顺序

### 第一步：确认搜索是否正常

```bash
ani-cli --source omofun111 "进击的巨人"
```

如果完全搜不到结果，优先检查搜索页访问是否受当前网络环境影响。

### 第二步：用 `debug` 模式确认是否已经拿到最终播放链接

```bash
ANI_CLI_PLAYER=debug ani-cli --source omofun111 -S 11 -e 1 "进击的巨人"
```

如果已经能看到最终 `m3u8` 或 `mp4`，说明站点解析链路已经打通。

### 第三步：切换播放器是否直连

默认直连：

```bash
ANI_CLI_PLAYER_NO_PROXY=1 ani-cli --source omofun111 "进击的巨人"
```

强制让播放器也走代理：

```bash
ANI_CLI_PLAYER_NO_PROXY=0 ani-cli --source omofun111 "进击的巨人"
```

## 结论

如果你遇到“不开代理搜不到，开代理又播不了”的情况，优先把它理解成网络出口兼容问题，而不是简单的播放器故障。

当前版本增加的直连播放策略，就是专门为这类场景准备的。
